# 性能优化

### 性能监控与核心指标

#### 性能优化方式（必看）

1. 加载性能优化：

* 资源加载-文件压缩、图片压缩、代码压缩
* 减少网络请求次数-雪碧图、字体优化、代码分割
* 利用缓存-Http缓存、使用本地缓存、CDN加速

2. 渲染性能优化：

* 避免阻塞渲染-关键资源优先加载、异步加载、懒加载
* 减少重绘和重排-批量操作DOM、避免频繁修改样式属性、虚拟列表
* 高效渲染逻辑：防抖节流、避免不必要的渲染

3. 进阶优化：

* 服务端渲染
* 预加载资源
* web workers
* 监控与分析

| **优化方向** | **具体方法** | **核心目标/原理** | **备注/收益** |
| --- | :--- | :--- | --- |
| **资源优化** | 代码分割 (Code Splitting)  | 按路由/组件拆分，仅加载首屏必需代码 | Vue 使用 `() => import()` ，React 使用 `React.lazy`  + `Suspense` |
| | 压缩与 Tree-shaking  | 移除未使用代码，压缩 JS、CSS | 使用 Terser、css-minimizer-webpack-plugin 等工具 |
| | 按需引入 UI 库  | 避免引入整个组件库，只引入需要的组件 | 如 Element Plus, Ant Design 都支持按需引入 |
| | 图片优化  | 使用 WebP/AVIF 格式，压缩体积，首屏图片数量控制（如不超过3张） | 非首屏图片使用懒加载 (`loading="lazy"`)  |
| **加载策略** | CDN 分发  | 利用边缘节点加速静态资源（JS、CSS、图片）加载 | 配置 `publicPath`  (Vue) 或 `homepage`  (React create-react-app)  |
| | 资源预加载/预连接  | 使用 `<link rel="preload">`  提前加载关键资源，`<link rel="preconnect">`  提前建立连接 | 优先预加载关键字体、首屏大图、核心 JS/CSS  |
| | 异步/延迟加载 JS  | 使用 `async`  或 `defer`  属性避免 JS 阻塞渲染 | 非关键 JS 可放在 `<body>`  底部或动态导入  |
| | HTTP/2 或 HTTP/3  | 支持多路复用，减少连接开销 | 需服务器支持 |
| **渲染优化**   | 服务端渲染 (SSR) / 静态站点生成 (SSG)  | 服务器直接返回渲染好的 HTML，减少浏览器端渲染压力 | Vue 可用 Nuxt.js，React 可用 Next.js  |
| | 骨架屏 (Skeleton Screen)  | 数据加载前显示占位图形，减少白屏感知 | 可通过专用插件（如 `vue-skeleton-webpack-plugin`）或手动编写  |
| | 内联关键 CSS (Critical CSS) | 将首屏渲染所需 CSS 直接内联到 HTML 的 `<style>`  标签中，避免外部 CSS 文件请求阻塞渲染 | 可使用 `critters-webpack-plugin`  等工具自动提取和内联  |
| **缓存策略** | 强缓存与协商缓存  | 对静态资源（JS、CSS、图片）设置长期缓存（如 `Cache-Control: max-age=31536000` ） | 配合文件名哈希（如 `app.[hash].js` ）实现缓存更新  |
| | Service Worker  | 缓存关键资源，支持离线访问 | 可通过 Workbox 或 Vue/React 的 PWA 插件实现  |
| **协议与传输优化** | Gzip/Brotli 压缩  | 服务器开启压缩，显著减少文本资源（HTML、CSS、JS）传输体积 | 对 JS/CSS 压缩率可达 70%+  |
| **其他** | 字体优化  | 使用 `font-display: swap`  避免字体加载阻塞文本显示，或预加载关键字体 | 也可考虑使用系统默认字体  |
| | 优化 JavaScript 执行  | 避免长任务阻塞主线程，复杂计算放入 Web Worker |  |
| | 客户端协助 (针对 App 内 WebView) | WebView 预创建  、资源离线化  、数据预取  | 需与客户端开发协作  |

#### 性能优化核心指标及量化方法（必看）

1. 加载性能：TTFB，FPR，FCP      请求到首次有效内容渲染时间，白屏时间
2. 交互性能：TTI，FPS，交互延迟   用户能否流程完成操作
3. 资源效率：LCP，FID，资源体积 页面加载速度和用户首次交互的延迟
4. 网络效率：首字节到达时间，TCP连接耗时，重定向次数，请求成功率   网络传输质量，太低要检查基础设施

| **指标名称 (英文全称)** | **中文名称** | **核心目标 (衡量什么)** | **优秀标准 (目标值)** | **主要优化方向** |
| :--- | :--- | :--- | :--- | :--- |
| **LCP (Largest Contentful Paint)** | **最大内容绘制** | 页面中最大内容元素（如图片、标题）在视口中渲染完成的时间 | ≤ 2.5 秒  | 优化关键资源加载、服务器响应、缓存策略  |
| **FID (First Input Delay)** | **首次输入延迟** | 用户首次与页面交互（如点击按钮）到浏览器实际响应的延迟 | ≤ 100 毫秒  | 减少 JavaScript 主线程阻塞，拆分长任务  |
| **CLS (Cumulative Layout Shift)** | **累积布局偏移** | 页面生命周期内所有意外布局偏移的累计分数 | ≤ 0.1  | 为媒体元素设置尺寸，预留动态内容空间  |
| **FCP (First Contentful Paint)** | **首次内容绘制** | 浏览器首次渲染出任何文本、图像、canvas 等内容的时间点 | ≤ 1.8 秒（桌面端） | 减少关键渲染路径，优化资源加载  |
| **TTI (Time to Interactive)** | **可交互时间** | 页面完全达到可用状态，能可靠响应用户交互的时间点 | ≤ 5 秒（移动端）/ ≤ 3 秒（桌面端） | 减少 JavaScript 执行时间，代码分割  |
| **TTFB (Time to First Byte)** | **首字节时间** | 浏览器发起页面请求到接收到服务器返回第一个字节数据时间 | ≤ 200ms  | 优化服务器响应，使用 CDN，启用缓存   |

#### DCL

`DOMContentLoaded`** **事件在浏览器已完全加载和解析 HTML 文档，并构建好 DOM 树时触发。此时不一定需要等待样式表、图片或子框架等外部资源加载完成。

**重要意义：**

DCL 时间标志着页面内容已准备就绪，用户可以开始与页面进行交互，它是前端性能优化的一个关键指标，较短的 DCL 时间意味着更快的页面可交互速度。

**影响因素与优化：**

* HTML 文档大小：文件越大，解析时间越长。
* 同步 JavaScript 脚本：会阻塞 DOM 解析，显著延迟 DCL。应尽量使用 `async` 或 `defer` 属性异步加载脚本。
* CSS：虽然通常不会直接阻塞 DOM 解析，但如果 JavaScript 需要等待 CSSOM 构建完成，则会间接影响 DCL。
* 优化策略：压缩 HTML、异步加载 JavaScript、优化关键渲染路径、使用 CDN 等。

##### 优化措施

前端优化 (针对 ~1200ms)

| **优化方向** | **具体措施** | **预期收益/目标** |
| :--- | :--- | :--- |
| **1. 优化 LCP 元素加载** | - **图片优化**：对 LCP 元素图片进行压缩（如 WebP/AVIF 格式转换 ）、预加载 (`<link rel="preload" as="image">` )。   - **优先级提示**：使用 `fetchpriority="high"` 提示浏览器优先加载 LCP 资源。 | 减少 LCP 元素加载时间，直接提升 LCP。 |
| **2. 减少 JS/CSS 影响** | - **代码分割/懒加载**：使用 `import()`动态导入或框架路由懒加载 ，减少首屏 JS 体积。   - **移除未使用代码**：利用打包工具 Tree-shaking。   - **内联关键 CSS**：将首屏所需 CSS 内联到 HTML 中，减少请求。 | 减少渲染阻塞，加快首屏渲染。 |
| **3. 提升资源加载效率** | - **CDN 加速**：静态资源部署到 CDN   - **HTTP/2 或 HTTP/3**：利用多路复用提升加载效率。   - **浏览器缓存**：设置合理的 `Cache-Control`策略  | 缩短资源下载时间，减少网络延迟。 |
| **4. 预连接/预解析** | - 使用 `<link rel="preconnect">`或 `<link rel="dns-prefetch">`提前建立到重要第三方源的连接 | 减少建立连接所需的时间。 |

后端优化 (针对 ~800ms)

| **优化方向** | **具体措施** | **预期收益/目标** |
| :--- | :--- | :--- |
| **1. 降低 TTFB** | - **服务器与基础设施**：升级服务器配置；应用和数据库部署在同一可用区以减少网络延迟；使用更快的 Web 服务器（如 Nginx）。   - **缓存**：对接口响应实施缓存策略（如 Redis 缓存），减少重复计算和数据库查询 | 缩短服务器响应时间。 |
| **2. 优化数据处理逻辑** | - **数据库优化**：为慢查询添加索引；避免 `SELECT *`，只取必要字段；考虑分库分表   - **代码优化**：避免循环嵌套；引入异步处理或并行计算（如多线程）。   - **缓存应用**：缓存常用数据，避免频繁查询数据库 | 减少服务器处理时间。 |
| **3. 减少数据传输量** | - **数据压缩**：启用 Gzip 或 Brotli 压缩   - **分页**：对列表数据实施分页   - **字段过滤**：接口只返回前端需要的字段。 | 减小响应体大小，缩短传输时间。 |

前后端协作优化

***数据预取/服务端渲染 (SSR)**：对于首屏关键数据，可以考虑在服务器端直接内嵌到 HTML 中（SSR），避免客户端再次请求接口，这能显著提升 LCP。
***接口聚合 (BFF层)**：如果首屏需要调用多个接口，可以考虑在后端设置一个 BFF (Backend for Frontend) 层来聚合这些数据，减少前端的请求数

#### 谈谈对于重构的理解

网站重构:在不改变外部行为的前提下，简化结构、添加可读性，而在网站前端保持一致的行为。也就是说是在不改变U的情况下， 对网站进行优化， 在扩展的同时保持一致的UI

对于传统的网站来说重构通常是:

* 表格( table )布局改为 DIV+CSS
* 使网站前端兼容于现代浏览器(针对于不合规范的CSS如对Im6有效的)
* 对于移动平台的优化针
* 对于 SEO 进行优化

### 渲染性能优化

#### 虚拟列表（必看）

1. **定高虚拟列表**：所有列表项高度固定。实现最简单，性能最高。通过 `scrollTop` 直接计算 `startIndex` 和 `endIndex`，并通过 `translateY` 进行偏移。
2. **不定高虚拟列表**：列表项高度不定但已知（或可预估）。实现较复杂。

***预估高度**：初始化时使用一个预估高度计算总高度和位置。
***测量与缓存**：列表项渲染后，通过 `getBoundingClientRect()` 获取实际高度并缓存，动态更新总高度和后续项的位置。
***二分查找**：滚动时，根据缓存的位置信息，使用二分查找快速确定 `startIndex`**10**。

3. **动态高度虚拟列表**：列表项高度可能动态变化（如内容折叠/展开）。实现最复杂，性能开销最大。

* 需要监听列表项高度的变化（如 `ResizeObserver`）。
* 一旦某个项高度变化，需重新计算其后所有项的位置和总高度。
* 务必做好防抖和缓存，避免频繁重排重绘。

##### 实现虚拟列表的关键步骤

1. **基础结构与样式**：

```html
<!-- 滚动容器 -->
<div class="viewport" style="height: 400px; overflow-y: auto;">
  <!-- 占位容器，用于模拟总高度，确保滚动条正确 -->
  <div class="scroll-phantom" :style="{ height: totalHeight + 'px' }"></div>
  <!-- 实际渲染内容的容器，使用绝对定位或变换 -->
  <div class="render-list" :style="{ transform: `translateY(${offset}px)` }">
    <!-- 循环渲染 visibleData 中的项 -->
    <div v-for="item in visibleData" :key="item.id" class="list-item">
      {{ item.content }}
    </div>
  </div>
</div>

```

```css
.viewport {
  position: relative; /* 或 absolute */
  overflow-y: auto;
}
.scroll-phantom {
  position: absolute;
  top: 0;
  left: 0;
  right: 0;
}
.render-list {
  position: absolute;
  top: 0;
  left: 0;
  right: 0;
}
.list-item {
  /* 你的列表项样式 */
}

```

2. **计算可视区域数据**：在滚动事件处理函数中，根据当前的 `scrollTop`、`viewportHeight` 和 `itemHeight`（或缓存的高度信息）计算 `startIndex`、`endIndex` 和 `offset`，并更新 `visibleData`。
3. **监听滚动事件**：为 `viewport` 容器添加 `scroll` 事件监听器，触发上述计算逻辑。**务必使用节流 (throttle) 或 **`requestAnimationFrame` 优化性能，避免频繁计算

##### 性能优化与常见问题处理

1. **缓冲区 (Buffer)**：在可视区域上下额外渲染一定数量的项（如各多渲染 5 项），快速滚动时减少白屏

```javascript
const buffer = 5; // 缓冲区大小
const startIndexWithBuffer = Math.max(0, startIndex - buffer);
const endIndexWithBuffer = Math.min(data.length, endIndex + buffer);

```

2. **减少重排重绘**：

* 使用 `transform: translateY()` 进行偏移，而非改变 `top`/`margin`，以利用硬件加速**3**。
* 避免在滚动事件中进行可能导致重排的操作。

3. **滚动事件优化**：使用 `requestAnimationFrame` 或节流函数控制滚动事件触发频率
4. **列表项复用与缓存**：对于复杂列表项，使用 `React.memo`、`Vue` 的 `keep-alive` 或 `shouldComponentUpdate` 避免不必要的重渲染
5. **异步加载与占位符**：结合懒加载，数据未加载时显示骨架屏占位符，提升体验
6. **动态高度处理的优化**：

***预估高度**：初始化时使用预估高度，待项渲染完成后再测量并更新缓存和布局。
***批量更新**：避免单个高度变化立即更新整体，可收集变化后批量处理。

7. **使用 Web Worker**：将大量数据的计算（如排序、过滤）移至 Web Worker，避免阻塞主线程

##### 适用场景

1. 长列表渲染：如电商商品列表、文件管理器。
2. 无限滚动 (Infinite Scroll)：如社交媒体信息流。
3. 大数据可视化：如日志查看器、大型表格。
4. 聊天应用：消息历史记录。
5. 移动端复杂列表：性能受限的设备。

#### 虚拟列表和懒加载的区别

| **特性** | **虚拟滚动 (Virtual Scrolling)** | **懒加载 (Lazy Loading)** |
| :--- | :--- | :--- |
| **核心思想** | 只渲染**可视区域内**的列表项，通过动态计算和更新来模拟完整列表 | **延迟加载**资源（如图片、数据），直到它们需要被显示时 |
| **优化目标** | 减少 **DOM 节点数量**，降低渲染压力，解决长列表滚动性能问题 | 减少**初始请求数**和**资源加载量**，加快页面初始加载速度 |
| **数据状态** | 通常**一次性获取**所有数据（或大分块），在前端进行截取和渲染 | 数据通常**按需分块加载**，滚动到底部时触发新数据请求 |
| **滚动条行为** | 滚动条**反映完整数据集的长度**，行为真实 | 滚动条长度随新内容加载而**动态变化** |
| **实现复杂度** | **较高**，需计算可视区域、维护滚动位置、处理动态高度等 | **相对较低**，主要监听滚动事件或使用 Intersection Observer API |
| **典型应用场景** | 大型数据列表、表格、聊天记录、日志文件等任何需要展示大量行数据的界面 | 图片画廊、商品列表、新闻信息流、评论分页等 |
| **主要技术手段** | 计算 `startIndex`和 `endIndex`，使用 `transform`进行偏移，创建占位元素 | 监听 `scroll`事件或使用 `IntersectionObserver`，替换 `data-src`为 `src` |

#### 虚拟滚动怎么用InterSection Oberver做

1. 计算视口高度：确定当前视口能容纳的元素数量
2. 创建占位元素：创建一个高度等于整个列表总高度的占位元素，用于模拟滚动条的高度
3. 监听滚动事件：通过 IntersectionObserver 监听目标元素是否进入视口
4. 动态渲染：根据滚动位置动态渲染视口内的元素

#### 长列表项渲染优化（必看）

从**原生 HTML/JavaScript** 和 **Vue.js** 两个角度

##### 原生 HTML/JavaScript 优化细节

1. **虚拟滚动 ，只渲染可视区域内的列表项**，并通过一个占位元素来模拟整个列表的滚动容器，从而保持正确的滚动条比例

**基本原理**：监听滚动容器的 `scroll`事件，根据滚动位置动态计算当前应该显示哪些列表项（`startIndex`和 `endIndex`），并只将这些项插入到 DOM 中。同时，通过 `transform: translateY(offset)`来调整渲染项的位置，使其看起来是连续的 **关键代码片段（简化）**：

```javascript
const container = document.getElementById('container');
const content = document.getElementById('content');
const itemHeight = 50; // 假设每项高度固定
const totalItems = 10000;
let startIndex = 0;

// 设置容器总高度，模拟滚动范围
content.style.height = `${totalItems * itemHeight}px`;

function renderVisibleItems() {
  const scrollTop = container.scrollTop;
  startIndex = Math.floor(scrollTop / itemHeight);
  const endIndex = startIndex + Math.ceil(container.clientHeight / itemHeight);

  // 清空当前显示项
  content.innerHTML = '';

  const fragment = document.createDocumentFragment();
  for (let i = startIndex; i <= endIndex; i++) {
    const item = document.createElement('div');
    item.className = 'item';
    item.textContent = `Item ${i + 1}`;
    item.style.position = 'absolute';
    item.style.top = `${i * itemHeight}px`; // 定位
    fragment.appendChild(item);
  }
  content.appendChild(fragment);
}

container.addEventListener('scroll', renderVisibleItems);
// 初始渲染
renderVisibleItems();

```

**优化点**：使用 `requestAnimationFrame`或 `requestIdleCallback`对滚动事件进行调度，避免过于频繁的渲染；设置**缓冲区**，提前渲染一些可视区域之外的项目，以减少快速滚动时的白屏

2. **分页与懒加载 **

**分页**：是最直观的方法。将数据分成若干页，每次只请求和渲染一页

**懒加载（无限滚动）**：监听滚动位置（例如接近列表底部时），动态加载下一页数据并追加到当前列表尾部,可以使用 `Intersection Observer API`来更高效地实现

**注意**：务必对滚动事件使用**防抖 (debounce)** 或**节流 (throttle)**，避免性能开销

3. **文档片段 (DocumentFragment)**：在需要批量添加多个 DOM 元素时，可以先将其附加到 `DocumentFragment`上，最后再一次性将 `DocumentFragment`插入到真实 DOM 中。这能减少重排次数

2. **减少响应式开销**：Vue 会对数据对象进行递归的响应式处理，对于纯展示的超大列表，这可能是不必要的开销。

**使用 **`Object.freeze()`：冻结数据对象，Vue 将不会使其变为响应式

```javascript

    return {
      list: []
    }
  },
  async created() {

    this.list = Object.freeze(data); // 冻结数据，避免响应式化
  }
};

```

**使用函数式组件**：对于非常简单的列表项，可以将其定义为函数式组件,它们没有实例，渲染开销更低

```vue

    <div v-for="item in props.items" :key="item.id" class="item">
      {{ item.text }}
    </div>
  </template>

```

3. **使用 **`v-memo`**进行记忆化 (Vue 3.2+)**：这个指令可以缓存一部分模板树，仅在依赖项发生变化时才会更新。对于大型列表中变化不频繁的部分非常有用

```vue

  {{ item.name }} <!-- 只有当 item.id 变化时才会重新渲染 -->
</div>

```

4. **避免内联计算和 Methods**：在 `v-for`中避免使用内联计算方法或调用方法，这会在每次重渲染时都执行。优先使用**计算属性 (computed properties)**

```vue
<!-- 避免 -->

  {{ calculateSomething(item) }}
</div>

  <!-- 推荐 -->
  <template>
  <div v-for="item in computedList" :key="item.id">
  {{ item.computedValue }}
</div>
  </template>
  <script>
  export default {
    computed: {
      computedList() {
        return this.list.map(item => ({
          ...item,
          computedValue: someExpensiveCalculation(item)
        }));
      }
    }
  }
</script>

```

5. **合理使用 **`key`：在 `v-for`中始终提供唯一且稳定的 `key`，这能帮助 Vue 最大程度地复用已有的 DOM 节点

##### 数据层面优化

数据层面的处理是解决性能问题的根本，目标是减少需要渲染的数据点数量。

**数据降采样 (Downsampling)**：这是最核心的优化手段。通过算法在保留数据整体趋势（如峰值、谷值、拐点）的前提下，大幅减少数据点数量。ECharts 折线图内置了 `sampling`配置项，可设置为 `'average'`（取桶内平均值）、`'max'`（取最大值）、`'min'`（取最小值）或 `'lttb'`（Largest-Triangle-Three-Buckets，能较好保留趋势和极值）

```javascript
option = {
  series: [{
    type: 'line',
    data: largeData, // 你的10万条原始数据
    sampling: 'lttb', // 开启内置降采样策略
    // ... 其他配置
  }]
};

```

**数据分层与按需加载**：不要一次性加载和渲染所有数据。初始只加载一个数据概览（如聚合后的数据）。当用户与图表交互（如缩放、拖动）时，再动态加载更精细的细节数据

**后端数据预处理**：在后端对数据进行聚合（如按时间粒度求和、求平均）、过滤无效值或噪音数据，再传输给前端，从前端源头减少数据量

优化渲染过程可以提升浏览器绘制图表的效率。

**选择高效的渲染器**：ECharts 默认使用 Canvas 渲染器，其在处理大数据量时性能通常优于 SVG

* 对于极大数据量（如数十万以上）或 3D 图表，可以考虑使用 **WebGL** 渲染器 (`echarts-gl`扩展) 以利用 GPU 进行并行加速渲染**13**。

##### 交互层面优化

优化交互体验可以确保用户在操作大型数据集时仍能获得流畅的响应。

**优化提示框 (Tooltip) 性能**：`tooltip.trigger`设置为 `'item'`而非 `'axis'`，因为后者会计算并渲染所有系列在同一位置的数据，在大数据量下计算量很大,可以限制 `tooltip`的触发方式为 `'click'`而非默认的 `'mousemove'`

#### 什么是重排和重绘？如何减少它们？（必看）

**重排**

***定义**：当 DOM 结构发生变化，导致浏览器需要**重新计算元素几何属性**（位置、大小）并更新渲染树的过程
***触发条件**：
  * 添加/删除 DOM 元素
  * 元素尺寸、位置、边距改变（width、height、margin、padding）
  * 窗口大小改变、字体大小改变
  * 获取布局信息（offsetTop、scrollTop、getComputedStyle 等）
***性能消耗**：非常**高**，会影响整个文档流

**重绘**

***定义**：当元素样式改变**但不影响布局**（如颜色、背景、透明度等），浏览器只需要重新绘制受影响元素的过程
***触发条件**：颜色、背景、边框样式、visibility、opacity 等改变
***性能消耗**：相对**较低**，只涉及绘制过程

***

**优化策略 **

***批量 DOM 操作**：
  * 使用 `documentFragment`进行批量添加
  * 隐藏元素（`display: none`）→ 修改 → 显示
***避免触发同步布局事件**：
  * 避免在循环中读取会触发重排的属性（offsetTop 等）
  * 使用 `requestAnimationFrame`优化动画
***CSS 优化**：
  * 使用 `transform`和 `opacity`实现动画（触发 GPU 加速）
  * 使用 `class`批量修改样式，而非逐条修改
  * 避免使用 `table`布局（一个小改动可能触发整个表格重排）
***分离读写操作**：
  * 先读取所有需要的布局属性，再进行修改
***使用定位属性**：
  * 对动画元素使用 `position: absolute/fixed`，使其脱离文档流

**优化原则**

优先使用 transform 和 opacity，尽可能利用 GPU 加速

**具体优化策略**

1. **使用 CSS3 动画替代 JavaScript 动画**：
   * CSS 动画在合成层进行，不触发重排和重绘
   * 使用 `transform`（位移、缩放、旋转）、`opacity`实现动画
2. **启用 GPU 加速**：

```css

  transform: translateZ(0); /* 或 translate3d(0,0,0) */
  will-change: transform;   /* 提前告知浏览器 */
}

```

```

- 将动画元素提升到独立合成层，避免影响其他元素

```

3\. **使用 **`requestAnimationFrame`：
\- 相比 `setTimeout/setInterval`，`requestAnimationFrame`与浏览器刷新率同步
\- 避免丢帧，动画更流畅

### 资源加载与网络优化

#### InterSection Observer会不会引发性能问题

1. **观察大量元素且未及时清理**：观察成千上百个元素（如超长列表），并且元素进入视口后没有用 `unobserve()` 解除观察，观察器内部维护的条目会越来越多。

**优化建议**：一旦元素完成触发，立即调用 `observer.unobserve(entry.target)` 移除观察。

2. **回调函数执行昂贵操作**：IO API的回调本身不会引起性能问题，但**如果在回调函数中执行复杂的同步JavaScript计算、频繁操作DOM**，就会阻塞主线程。

**优化建议**：保持回调函数轻量。将耗时操作放入 `requestIdleCallback`、`setTimeout` 中异步执行，或交由 Web Worker 处理。

3. **配置过于“敏感”**：设置 `threshold: [0, 0.01, 0.02, ..., 1]` 这样的多级阈值，或在 `rootMargin` 中设置过大的预加载范围，可能导致回调函数被非常频繁地调用

**优化建议**：根据实际需求设置合理的阈值和rootMargin。例如，图片懒加载设置 `threshold: 0.01` 即可，无限滚动监听一个特定触发点（`threshold: 1`）。

4. **使用Polyfill的代价**：在不支持IO API的旧版浏览器中，Polyfill通常会降级为基于 `scroll` 事件和 `getBoundingClientRect()` 的模拟实现，这会重新引入传统方案的那些性能问题。

**优化建议**：评估目标用户群体的浏览器使用情况，权衡是否真的需要Polyfill。

#### InterSection Oberver除了首页优化还能做些什么

1. 核心原理是通过异步回调机制来监测目标元素与视窗的交叉状态、当目标元素进入或离开视口时，会触发回调函数，从而执行相应的操作
2. 这种异步、低开销的方式，使得它非常适合用于实现懒加载等需要监听元素可见性变化的场景。
3. 主要作用：**图片懒加载**、**无限滚动加载**、**广告曝光统计**、**交互动画触发**、**关键内容阅读进度追踪**
4. 配置参数

***root**：用于观察目标元素交叉的根元素，通常为视口（null）。
***rootMargin**：根元素的外边距，可以用来扩展或缩小视口。
***threshold**：触发回调的交叉阈值，可以是一个数字（0 到 1 之间），也可以是一个数组。

#### 图片懒加载的三方库实现

| **库名称** | **特点/优势** | **文件大小 (压缩后)** | **兼容性** | **资源类型支持** |
| :--- | :--- | :--- | :--- | :--- |
| **lozad.js** | 基于现代 Intersection Observer API，高性能、轻量级、可配置 | ~1 KB | IE11+ | 图片、视频、iframe  |
| **lazysizes** | 功能全面，纯 JavaScript 支持多种加载策略和响应式图片 | ~10 KB | IE8+ | 图片、视频、iframe |
| **vanilla-lazyload** | 轻量级，无第三方依赖，使用简单 | ~1 KB | IE9+ | 图片 |
| **unlazy** | 基于原生 `loading="lazy"`支持 BlurHash/ThumbHash 模糊预览 自动计算 `sizes`、SEO友好 | - | 现代浏览器 | 图片 (`<picture>`标签) |
| **yall.js** | SEO友好，在可用时使用 Intersection Observer，兼容 CSS 背景图片和 `<video>`、元素海报  | - | 所有现代浏览器 | `<video>` 元素海报、CSS背景图像（v4版） |
| **vue-lazyload** | Vue 专用，提供 `v-lazy`指令，支持加载中和加载错误占位图 | - | - | 图片（Vue生态） |

选择时可以考虑以下几点：

1. **项目技术栈**：Vue 项目可优先考虑 `vue-lazyload`**4**；React、Angular 或纯 JavaScript 项目可选择 `lozad.js`、`lazysizes`、`vanilla-lazyload` 或 `unlazy`**56**。
2. **兼容性要求**：需支持旧版浏览器（如 IE10 及以下），`lazysizes` 是不错的选择**6**。主要面向现代浏览器，可优先选择基于 Intersection Observer API 的 `lozad.js` 或 `unlazy`**35**。
3. **功能需求**：
   * 只需基本图片懒加载，`vanilla-lazyload` 或 `lozad.js` 都很轻量**6**。
   * 需要**模糊预览效果**，`unlazy` 支持 BlurHash/ThumbHash**5**。
   * 需要处理**视频懒加载**或 **CSS 背景图片**，`yall.js` 或 `lazysizes` 可能更合适**36**。
   * 需要**响应式图片**（`srcset`、`sizes`），`lazysizes` 和 `unlazy` 提供良好支持**15**。
4. **性能与体验**：关注库是否轻量以及对性能的影响。大多数现代库都已优化。

#### LazyLoad加载组件时机

| **触发时机 ** | **简要说明 ** | **典型场景 ** |
| :--- | :--- | :--- |
| **进入可视区域**  | 当组件即将进入用户浏览器可视窗口时加载。 | 长列表、图片流、无限滚动等 |
| **路由导航** (针对路由懒加载)  | 当用户点击链接或改变URL，匹配到懒加载的路由组件时加载。 | Vue Router 或 React Router 中使用 `() => import('./MyComponent.vue')` 或 `React.lazy()` |
| **代码逻辑触发** (手动控制)  | 在特定的用户交互或程序逻辑满足条件时，通过代码手动触发加载。 | 点击按钮后加载弹窗组件、某个状态变为 true 时加载特定功能模块 |
| **预加载** (提前加载)  | 在浏览器空闲时或预测用户行为（如鼠标悬停在链接上）时，提前加载组件。 | 使用 `requestIdleCallback` 或在 `Link`组件的 `onMouseEnter`事件中预加载 |
| **组件首次渲染** (Suspense 内)  | 对于被 `Suspense`  包裹的 `React.lazy`  组件，在其首次渲染时就会开始加载。 | 在 React 应用中，使用 `<Suspense fallback={<div>Loading...</div>}><MyLazyComponent /></Suspense>` |

#### 讲讲请求合并

***时间窗口合并**：设置一个固定的时间窗口（例如50ms）。在此期间到达的所有请求会被收集起来，窗口结束时统一发送。
***数量阈值合并**：当收集到的请求数量达到预设的阈值（例如100个）时，立即触发批量请求发送。这类似于电梯，人满就出发，适合可以进行批处理的场景
***混合策略**：结合时间和数量两个维度，满足任一条件（时间到或数量够）就触发发送。这是生产环境中最常用的策略，能在实时性和效率间取得较好平衡

##### 使用 `Promise.all`进行请求并行化

`Promise.all`允许你并行发起多个请求，并等待所有请求完成。这虽然**没有减少实际的请求数量**，但通过并行化显著缩短了总等待时间，特别适合多个独立请求的场景

```javascript
// 假设有三个独立的API请求
const fetchUserInfo = fetch('/api/user/info').then(res => res.json());
const fetchUserOrders = fetch('/api/user/orders').then(res => res.json());
const fetchProducts = fetch('/api/products').then(res => res.json());

Promise.all([fetchUserInfo, fetchUserOrders, fetchProducts])
  .then(([userInfo, userOrders, products]) => {
    // 所有数据都获取成功，在这里处理业务逻辑
    console.log(userInfo, userOrders, products);
  })
  .catch(error => {
    // 其中任何一个请求失败，都会进入这里
    console.error('有一个请求出错了', error);
  });

```

#### 限流的风险

##### 用户体验下降

如果限流时间间隔设置得过长（比如 500ms 以上），用户快速滚动时可能会感到“卡顿”或“内容加载慢”

##### 重复加载或数据错乱

* 如果限流函数内部没有做状态管理（比如“正在加载”的标记），用户可能在短时间内多次触发加载，导致重复请求。
* 数据返回顺序错乱：如果网络请求时间不确定，先发的请求可能后返回，导致内容顺序错乱。

**应对建议：**

* 增加 **loading 状态锁**，确保同一时间只有一个加载请求。
* 如果数据顺序很重要，可以考虑 **取消之前的请求** 或对返回数据做排序处理。

##### 内存泄漏或事件监听未清理

* 在单页应用（SPA）中，如果组件卸载时没有移除滚动事件监听，会导致内存泄漏。
* 限流函数如果引用了外部变量（如组件状态），闭包引用可能导致垃圾回收失败。

**应对建议：**

* 在组件卸载时，主动移除事件监听
* 使用 `useEffect` 的清理函数（在 React 中）或 `beforeUnmount` 钩子（在 Vue 中）来清理事件和定时器

##### 移动端兼容性问题

* 移动端滚动事件（如 `touchmove`）的触发机制和 PC 不同，限流不当可能导致滚动不流畅。
* 某些浏览器（如 iOS Safari）在滚动时会有滚动惯性，限流逻辑可能无法正确捕获滚动结束。

**应对建议：**

* 在移动端，可以结合 `requestAnimationFrame` 来优化滚动事件，而不是单纯依赖 `setTimeout` 或时间戳。
* 使用 `passive: true` 事件监听选项，提升滚动性能：

##### 性能监控困难

* 限流后，滚动事件的实际触发次数变少，可能导致性能监控工具（如 Lighthouse、Web Vitals）无法准确捕捉到性能瓶颈。
* 如果限流逻辑封装得不好，可能难以排查滚动卡顿的具体原因。

#### 图片优化技术（必看）

**CSSSprite(雪碧图/精灵图):**

一种网页图片应用处理方式。它允许你将一个页面涉及到的所有零星图片都包含到一张大图中去，当访问该页面时，载入的图片就不会像以前那样一幅一幅地慢慢显示出来了。能很好地减少网页的http请求，从而大大的提高页面的性能

**SVGSprite:**

与 CSSSprite 类似,但是使用 SVG 格式。SVG 是矢量图形,放大不失真,而且文件体积通常较小。可以将多个 SVG 图标组合在一起,通过 标签引用需要的部分

**lconfont:**

将图标制作成字体文件,可以像使用文字一样使用图标。具有良好的可扩展性,支持样式修改,并且文件体积小。适合单色图标的使用场景

**Base64:**

将图片转换为 Base64 编码,直接嵌入到 CSS 文件中。可以减少 HTTP 请求,对于小图片特别有效。但会增加 CSS 文件体积,不适合大图片

这些技术各有优势,可以根据具体场景选择使用:

* 多个小图标建议使用 CSSSprite 或 SVGSprite
* 单色图标优先考虑 Iconfont
* 小型图片可以使用 Base64
* 需要高清晰度时推荐 SVGSprite

### 复杂计算与多线程优化

#### 讲讲web woker的详细信息（必看）

1. *Web Worker 的核心原理*

**独立线程**：Web Worker 在独立的线程中运行，与主线程完全隔离。这意味着它不会阻塞主线程的执行，也不会影响页面的渲染和用户交互。

**消息通信**：主线程和 Web Worker 之间通过 `postMessage` 和 `onmessage` 事件进行通信。主线程可以通过 `worker.postMessage()` 向 Worker 发送数据，Worker 也可以通过 `self.postMessage()` 将结果返回给主线程。

**无 DOM 访问**：Worker 线程无法直接访问 DOM、`window`、`document` 等对象，但可以使用 `navigator`、`location` 等对象。

2. *Web Worker 的类型*

**专用 Worker** (Dedicated Worker)：

由主线程创建，只能与创建它的主线程通信。

适用于需要独立执行的任务，如数据处理、计算密集型任务等。示例：`new Worker('worker.js')`

**共享 Worker** (Shared Worker)：

可以被多个脚本或页面共享，适用于需要跨多个脚本或页面协调工作的场景。

通过 `new SharedWorker('shared-worker.js')` 创建。使用 `port` 对象通信，`worker.port.postMessage()`。

**Service Worker**：

主要用于离线缓存、推送通知、后台同步等场景。

与 Web Worker 不同，Service Worker 通常用于 PWA（渐进式 Web 应用）中，提供更强大的离线功能。

3. *Web Worker 的使用场景*

**计算密集型任务**：图像处理、音视频编解码、物理模拟等需要大量计算的任务。

**后台任务**：定时任务（如定时检查更新）、文件处理（如大文件的分片上传或下载）。

**提高响应速度**：耗时任务（如大数据处理）放到 Worker 中执行，避免阻塞主线程，从而提高页面的响应速度。

**实时通信**：WebSocket 数据解析、实时数据处理等场景。

**Web Worker 的优缺点**

**优点**：

**不阻塞主线程**：Worker 线程与主线程并行执行，避免页面卡顿。

**性能提升**：多线程并行处理任务，提升应用性能。

**隔离环境**：Worker 崩溃不会影响页面，提高稳定性。

**缺点**：

**不能操作 DOM**：Worker 无法直接访问 DOM，无法直接修改页面内容。

**通信成本**：主线程和 Worker 之间只能通过消息传递数据，存在序列化和反序列化的开销。

**资源消耗**：创建和销毁 Worker 有一定的开销，不适合处理轻量级任务。

4. *Web Worker 的使用注意事项*

**同源限制**：Worker 脚本必须与主线程的脚本文件同源，否则会报错。

**文件限制**：Worker 无法读取本地文件（如 `file://` 协议），必须通过网络加载。

**错误处理**：主线程可以监听 Worker 的 `error` 事件，Worker 内部也可以监听 `error` 事件。

**生命周期管理**：Worker 一旦创建就会一直运行，直到被显式终止（`worker.terminate()`），避免资源浪费。

5. *Web Worker 与 Proxy、iframe 的对比*

**Proxy 代理**：

轻量级逻辑隔离，适合需要部分宿主权限的插件。

隔离级别较低，安全性中等。

**Web Worker**：

物理线程隔离，适合高安全要求或计算密集型任务。

安全性高，但通信成本较高。

**iframe**：

浏览器级进程隔离，适合完全不可信的第三方内容。

安全性极高，但资源消耗大，通信复杂。

#### webwoker可以操作dom元素嘛，能获取到window对象嘛

* Web Worker 不能直接操作 DOM 元素的原因*

1. **线程隔离性**\ Web Worker 运行在与主线程**完全隔离**的环境中。这意味着它不能直接访问主线程的 DOM 树，因为 DOM 是主线程的一部分。如果多个线程同时操作同一个 DOM 元素，可能会导致数据不一致、页面崩溃等问题。
2. **线程安全性**\ DOM 操作通常依赖于一些**非线程安全的全局对象**（如 `document`、`window` 等）。如果 Web Worker 直接操作这些对象，可能会引发不可预测的错误，甚至导致浏览器崩溃。
3. **W3C 规范限制**\ 根据 W3C 的规范，Web Worker **不能访问** `window`、`document`、`parent` 等对象。这是为了确保主线程的稳定性，避免因多线程操作导致的冲突。

* Web Worker 不能获取 *`_window_`* 对象的原因*

1. **运行环境不同**\ Web Worker 运行在独立的全局上下文中，其全局对象是 `WorkerGlobalScope`，而不是 `window`。这意味着 Web Worker 无法直接访问 `window` 对象及其相关属性和方法（如 `document`、`location`、`navigator` 等）。
2. **安全限制**\ `window` 对象包含了浏览器的许多敏感信息（如 `navigator`、`location` 等），如果 Web Worker 可以直接访问这些信息，可能会带来安全风险。因此，浏览器对 Web Worker 的访问权限进行了严格限制。

* Web Worker 能访问哪些对象？*

1. **全局对象 **`self`\ Web Worker 的全局对象是 `self`，它类似于 `window`，但功能有限。你可以通过 `self` 访问一些全局属性和方法，例如：
   * `self.postMessage()`：用于与主线程通信。
   * `self.onmessage`：用于监听主线程发送的消息。
   * `self.importScripts()`：用于加载其他脚本文件。
2. **网络请求**\ Web Worker 可以使用 `XMLHttpRequest` 或 `fetch` 发起网络请求，获取数据。
3. **定时器**\ Web Worker 支持 `setTimeout` 和 `setInterval`，可以用于定时执行任务。
4. **IndexedDB 和 Web Storage**\ Web Worker 可以访问 `IndexedDB` 和 `localStorage`，但需要注意，这些存储是主线程和 Worker 共享的，需要确保数据的一致性。
5. **WebSocket**\ Web Worker 可以通过 `WebSocket` 与服务器进行实时通信。
6. **Worker 内部的函数和类**\ Web Worker 可以使用 JavaScript 的内置函数和类，如 `Array`、`Object`、`Math` 等。

#### webworker的线程池复用逻辑

1. **线程预创建**\ Web Worker 线程池在初始化时会预先创建一定数量的 Worker 实例。这些 Worker 被存储在一个“空闲线程池”中，等待任务分配。
   * **优点**：避免了每次任务到来时都需要创建新线程的开销。
   * **缺点**：需要合理配置线程池大小，避免资源浪费或不足。
2. **任务分配机制**\ 当有新任务提交时，线程池会从空闲线程中选择一个 Worker 来执行任务。如果所有 Worker 都在忙碌，则任务会被放入任务队列中等待处理。
   * **任务队列**：用于存储待处理的任务，确保任务不会被丢弃。
   * **任务调度**：线程池通过调度算法（如 FIFO、优先级等）决定任务的执行顺序。
3. **线程复用**\ 一旦一个 Worker 完成当前任务，它会重新回到空闲线程池中，等待下一次任务分配。
   * **关键点**：Worker 在执行完一个任务后不会被销毁，而是继续等待新的任务，从而实现复用。
   * **实现方式**：Worker 在执行任务时会监听 `onmessage` 事件，当接收到新任务时，通过 `postMessage` 方法执行任务。
4. **任务执行流程**
   * Worker 接收到任务后，通过 `onmessage` 事件处理任务。
   * 任务处理完成后，Worker 通过 `postMessage` 将结果返回给主线程。
   * Worker 回到空闲线程池，等待下一次任务。
5. **资源管理**
   * **线程池大小**：根据系统资源和任务负载动态调整线程池大小，避免资源浪费或过载。
   * **任务拒绝策略**：当线程池已满且任务队列已满，线程池可以采取拒绝策略（如抛出异常、丢弃任务等）。

#### Service Worker 和 Web Worker 有什么区别？

| **对比维度** | **Web Worker** | **Service Worker** |
| --- | --- | --- |
| **定位** | 通用计算线程 | 网络代理 + 离线缓存 |
| **主要用途** | CPU 密集型计算（图像处理、加密解密、大数据计算） | 离线体验、资源缓存、推送通知、后台同步 |
| **生命周期** | 随页面创建和销毁 | 独立于页面，可长期运行（直到被手动停止） |
| **DOM 访问** | 不能访问 DOM | 不能访问 DOM |
| **网络请求** | 有限制（不能使用同步 API） | 可拦截和处理网络请求 |
| **存储访问** | 有限制（可使用 IndexedDB） | 完全访问（Cache API、IndexedDB） |
| **通信方式** | `postMessage`/ `onmessage` | `postMessage`/ 消息推送 / 缓存策略 |
| **注册位置** | 页面中注册 | 页面中注册，但独立运行 |
| **典型应用** | 1. 大数据排序/过滤   2. 图像/视频处理   3. 复杂算法计算 | 1. PWA（渐进式 Web 应用）   2. 离线访问   3. 资源缓存策略   4. 后台数据同步 |
