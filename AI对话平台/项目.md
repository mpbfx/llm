### 🎙️ 面试版（1分钟内语音回答）

面试官您好，关于 AI 对话中“上下文历史记忆”的实现，核心原理其实是因为**大模型本身是无状态的**，它不记得你之前说过什么，所以需要我们前端**在每次发请求时，把历史记录一起带给它**。

具体落地我分了三个步骤来做：
**1. 状态管理与持久化：** 我使用 Pinia 来统一管理聊天数据。以 `sessionId` 为键，把每个会话的 `message` 数组独立存储在 Store 中。并且通过插件（如 `pinia-plugin-persistedstate`）持久化到 LocalStorage，确保用户刷新页面记忆不丢失。
**2. 组装请求参数：** 每次发送消息前，前端会去 Pinia 的当前会话中捞取历史记录，把它们格式化成后端需要的数组结构（包含 `role: user/assistant` 和 `content`）。
**3. 滑动窗口策略（Token 优化）：** 因为模型有 Token 限制和计费成本，如果是上百轮的对话，我不可能全发过去。所以我实现了一个**滑动窗口机制**，比如只截取最近 5 到 10 轮的有效问答，再拼接上当前的 `System Prompt`（系统提示词），最终打包通过 SSE 发送给后端。

这就实现了兼顾大模型算力成本和聊天连贯性的上下文记忆。

### 📝 详细解析版（面临深度追问的准备）

在真实的业务场景下，实现上下文记忆不仅是“数组拼接”这么简单，面试官往往会考察你对性能、边界情况和成本的思考：

#### 1. 核心流程：如何将记忆“喂”给大模型
大部分 LLM 的 API（如 OpenAI 规范）期望接收到的输入是一个特定格式的数组：
```javascript
const requestContext = [
  { role: "system", content: "你是一个资深前端专家..." },  // 必须置顶的角色设定
  { role: "user", content: "什么是闭包？" },               // 历史对话1
  { role: "assistant", content: "闭包是..." },             // 历史对话2
  { role: "user", content: "怎么解决内存泄漏？" }        // 当前新问题
]
```
前端在 `sendMessage` 时，需要实时从 Pinia 里构建出这样一个数组上传。

#### 2. 滑动窗口：避免“记忆爆炸”（Token 限制与成本控制）
随着对话进行，上下文数组会越来越长。如果直接发送：
- **触发报错**：超过模型的最大上下文窗口（比如 8K / 128K Token）。
- **成本高昂**：每次请求都要为全部历史买单。
- **响应变慢**：模型处理超长文本的时间显著增加。

**解决方案（滑动窗口）：**
每次只取数组的**后 N 项**（比如最近的 10 条消息），代码逻辑类似：
```javascript
// 假设 messages 是当前会话的所有历史记录
let context = messages
  .filter(msg => msg.status === 'success') // 过滤掉发送失败的废消息
  .slice(-10)                              // 核心：只截取最近的 10 条
  .map(msg => ({ role: msg.role, content: msg.content }));

// 将全局系统设定插入到最前面
context.unshift({ role: "system", content: currentThemePrompt }); 
```

#### 3. 分支对话的隔离（数据结构设计）
用户可能同时开启了多个不同的聊天任务。在 Pinia 中不能只是一个一维的 `messages` 数组。
我们可以采用基于 **Map（或以 SessionID 为 key 的对象）** 的数据字典，把记忆相互隔离：
```javascript
// Pinia State 结构
state: () => ({
  activeSessionId: 'session_123',
  sessions: {
    'session_123': [{ id: 1, role: 'user', content: '...' }, ...],
    'session_456': [{ id: 1, role: 'user', content: '...' }, ...]
  }
})
```
切换左侧会话列表时，只要改变 `activeSessionId`，就能瞬间利用 `O(1)` 的复杂度切换到另一套由于闭包带来的完整记忆体系。

#### 4. 特殊场景：中途关联文件的上下文（多模态记忆）
简历中提到了大文件切片上传功能。对于带附件的对话，前端还需要把上传成功后服务端返回的 `fileId` 或文档解析后的文本，作为一种特殊的 `System Message` / 特殊结构一起塞入当前的上下文中去：
```javascript
// 带附件记忆的上下文构建
if (currentFileId) {
   context.push({ 
     role: "user", 
     content: `[Attached File ID: ${currentFileId}]\n 用户问题：请总结这份文档。` 
   });
}
```
