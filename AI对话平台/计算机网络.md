# 计算机网络

### HTTP协议演进

#### http1.1和2区别 优缺点（必背）

**1.⼆进制协议：**

HTTP/2 是⼀个⼆进制协议。

在 HTTP/1.1 版中，报⽂的头信息必须是文本（ASCII 编 码），数据体可以是文本，也可以是⼆进制。

HTTP/2 则是⼀个彻底的⼆进制协议，头信息和 数据 体都是⼆进制，并且统称为帧，可以分为头信息帧和数据帧。 帧的概念是它实现多路复⽤的 基 础。

**2.多路复用：**

HTTP/2 实现了多路复⽤，HTTP/2 仍然复⽤ TCP 连接，但是在⼀个连接⾥，客⼾端和 服务器都可以同时发送多个请求或回应，⽽且不⽤按照顺序⼀⼀发送，这样就避免了队头堵塞 的问题。

**3.数据流： **

HTTP/2 使⽤了数据流的概念，因为 HTTP/2 的数据包是不按顺序发送的，同⼀个连接⾥⾯连续的数据包，可能属于不同的请求。因此，必须要对数据包做标记，指出它属于哪个请 求。

HTTP/2 将每个请求或回应的所有数据包，称为⼀个数据流。每个数据流都有⼀个独⼀⽆⼆的 编 号。数据包发送时，都必须标记数据流 ID ，⽤来区分它属于哪个数据流。

**4.头信息压缩：**

HTTP/2 实现了头信息压缩，由于 HTTP 1.1 协议不带状态，每次请求都必须附上所有 信息。所以，请求的很多字段都是重复的，⽐如 Cookie 和 User Agent ，⼀模⼀样的内容，每 次请 求都必须附带，这会浪费很多带宽，也影响速度。HTTP/2 对这⼀点做了优化，引入了头信息压缩机制。一方面，头信息使用 gzip 或 compress 压缩后再发送；另⼀方面，客户端和服务器同时 维 护⼀张头信息表，所有字段都会存⼊这个表，⽣成⼀个索引号，以后就不发送同样字段了，只 发送索引号，这样就提高速度了。

**5.服务器推送： **

HTTP/2 允许服务器未经请求，主动向客户端发送资源，这叫做服务器推送。使⽤服 务器推送提前给客⼾端推送必要的资源，这样就可以相对减少⼀些延迟时间。这⾥需要注意的 是 http2 下服务器主动推送的是静态资源，和 WebSocket 以及使⽤ SSE 等⽅式向客⼾端发送即时数据的推送是不同的

#### http2解决了什么问题（必背）

1. 队头阻塞（Head-of-Line Blocking）

问题：HTTP/1.x 中，每个请求必须串⾏处理，前⼀个请求完成才能发送下⼀个，导致带宽浪费。

解决方案：HTTP/2 引⼊ 多路复⽤（Multiplexing），允许同⼀ TCP 连接上并行传输多个请求和响应，⽆需等待前⼀个请求完成。

2. 冗余头部开销

问题：HTTP/1.x 的头部（如  User-Agent、Cookie ）每次请求都会重复传输，浪费带宽。

解决方案：HTTP/2 采⽤ HPACK 压缩算法，将头部信息压缩为动态表，显著减少传输体积。

#### http2 仍然存在的局限性

1. 依赖 TCP 协议\
   问题：\
   TCP 的可靠传输机制（重传、顺序控制）在⾼丢包或⾼延迟⽹络（如移动⽹络）中可能导致额外延迟。\
   建⽴连接需经历三次握⼿（3-way handshake），耗时较⾼。\
   ⽰例场景：视频流传输中，TCP 可能因丢包频繁重传，影响实时性。
2. 移动端⽹络切换问题\
   问题：移动设备切换 Wi-Fi/蜂窝⽹络时，TCP 连接需重新建⽴，导致请求中断或超时。
3. ⽆法完全消除队头阻塞\
   问题：虽然 HTTP/2 ⽀持多路复⽤，但如果⼀个 TCP 流因丢包被阻塞，同⼀连接上的其他流也会被卡住（TCP 层⾯的队头阻塞）。

#### http3+udp

HTTP/3 是基于 **QUIC 协议**的全新传输层协议，运行在 UDP 上，而非 TCP。以下是其核心改进：

**1. 基于 UDP 的无连接传输**

***优势**：
  * **0-RTT 连接建立**：通过 QUIC 的加密握手（替代 TCP 三次握手），首次连接可快速完成。
  * **移动网络友好**：网络切换时，QUIC 可通过现有加密会话快速恢复连接，减少重传。
***对比 TCP**：UDP 不保证可靠传输，但 QUIC 在应用层实现了类似功能（如重传、顺序控制），兼顾效率和灵活性。

**2. 彻底消除队头阻塞**

***实现方式**：
  * QUIC 将每个请求/响应视为独立的流（Stream），不同流之间互不干扰。
  * 即使某个流因丢包被阻塞，其他流仍可并行传输。
***示例场景**：视频流和文件下载可以同时进行，互不影响。

**3. 改进拥塞控制**

***问题**：TCP 的拥塞控制算法（如 Cubic）在带宽高但延迟大的网络（如卫星通信）中表现不佳。
***解决方案**：QUIC 采用 **CUBIC** 的改进版本（如 QUIC 的 Reno）和基于延迟的拥塞控制，适应不同网络环境。

**4. 内置安全性**

***TLS 1.3 集成**：QUIC 将加密握手（TLS 1.3）集成到协议中，简化了安全传输流程，同时抵御中间人攻击。

#### 讲讲http缓存协议（必背）

**一、HTTP缓存的作用与必要性**

由于网络链路漫长且时延较高，直接从服务器获取资源成本较高。HTTP缓存机制允许客户端（如浏览器）存储已获取的资源，后续请求时优先使用本地缓存，从而减少网络传输，显著提升加载速度

**二、缓存控制的响应头字段**

HTTP响应头中的字段是控制缓存行为的关键，主要包括：

1. **Cache-Control**\ HTTP/1.1引入的字段，通过指令精确控制缓存行为。例如：
   * `no-store`：禁止缓存任何资源
   * `max-age`：设置缓存过期时间（如`max-age=3600`表示缓存1小时有效）
2. **其他字段**\ 如`Expires`（HTTP/1.0，指定绝对过期时间）和`ETag`（用于协商缓存，通过资源唯一标识判断是否更新）

**三、常见缓存策略**

1. **强制缓存**\ 浏览器直接根据`Cache-Control`或`Expires`判断缓存是否过期。若未过期，则直接使用本地缓存，无需请求服务器
2. **协商缓存**\ 当缓存过期或需验证时，客户端携带`ETag`或`Last-Modified`等字段向服务器请求确认资源是否更新。若未更新，服务器返回`304 Not Modified`，客户端继续使用缓存
3. **禁止缓存**\ 通过`Cache-Control: no-store`或`Pragma: no-cache`（兼容HTTP/1.0）确保资源不被缓存，适用于实时性要求高的场景

**四、实际应用场景**

***静态资源**（如图片、CSS、JS）：通常设置较长的`max-age`，利用强制缓存减少重复加载
***动态内容**：使用协商缓存，确保用户获取最新数据的同时避免不必要的请求

### 协议底层

#### tcp为什么要三次握手而不是两次（必背）

TCP（传输控制协议）使用三次握手（three-way handshake）过程来建立可靠的连接。这个过程之所以设计为三次而不是两次，主要是为了确保连接的可靠性和避免潜在的资源浪费。以下是几个关键原因：

1. **确认双方的接收和发送能力**：
   * 第一次握手：客户端发送一个SYN（同步序列编号）报文到服务器，并进入SYN\_SENT状态，等待服务器确认。
   * 第二次握手：服务器收到SYN报文后，会发送一个SYN+ACK（同步序列编号+确认应答）报文作为应答，并将连接状态设置为SYN\_RCVD。
   * 第三次握手：客户端收到服务器的SYN+ACK报文后，会向服务器发送一个ACK（确认应答）报文，并进入ESTABLISHED状态，服务器在收到ACK报文后也进入ESTABLISHED状态。

如果只有两次握手，客户端无法确认自己的发送能力是否正常，因为服务器可能收到了客户端的SYN，但客户端可能没有收到服务器的ACK。

1. **避免旧的连接请求建立连接**：
   * 如果使用两次握手，可能会出现这样的情况：客户端发送了一个连接请求，但是这个请求在网络中长时间延迟，客户端可能已经超时并重新发送了新的连接请求。如果服务器只进行两次握手，它可能会错误地认为延迟的请求是新的有效请求，从而建立不必要的连接。
   * 三次握手可以防止这种情况发生，因为即使旧的连接请求到达服务器，由于没有完成第三次握手，服务器也不会认为连接已经建立。
2. **同步序列号**：
   * 三次握手过程中，双方可以同步各自的初始序列号，这是TCP可靠传输的基础。序列号用于标识传输的字节流，确保数据按正确的顺序传输。
   * 如果只有两次握手，双方无法确保序列号的同步，这可能会导致数据传输的混乱。

总之，三次握手是TCP协议设计中的一个关键机制，它确保了连接的可靠性和数据传输的正确性。通过这个过程，双方都能确认对方的存在和能力，并且同步了序列号，从而避免了资源浪费和潜在的错误连接。

### 安全与加密

#### 运营商试怎么拦截非法请求的

运营商拦截非法请求通常涉及一系列的技术手段和政策规定，以下是一些常见的做法：

1. **DNS过滤**：
   * 运营商可以控制DNS服务器，当用户尝试访问某个网站时，DNS服务器不会返回该网站的IP地址，或者返回一个错误的IP地址，从而阻止用户访问。
2. **IP地址封锁**：
   * 运营商可以直接封锁特定的IP地址或者IP地址段，使得用户无法访问这些地址。
3. **深度包检测（Deep Packet Inspection, DPI）**：
   * 运营商可以使用DPI技术来分析网络流量，识别并拦截特定的数据包。例如，可以识别HTTP请求中的URL或内容，如果内容违反了规定，则可以丢弃或修改这些数据包。
4. **流量重定向**：
   * 当检测到非法请求时，运营商可以将用户的请求重定向到一个警告页面或者替代内容页面。
5. **内容过滤**：
   * 运营商可以在网络层面实施内容过滤，比如基于关键词、正则表达式或其他算法来识别并拦截包含非法内容的流量。
6. **URL过滤**：
   * 运营商可以维护一个黑名单或白名单，根据URL来决定是否允许访问。黑名单中的URL会被直接拦截。

#### https和http的区别，用了什么机制去保证安全（必背）

**区别**

HTTP 是超⽂本传输协议，信息是明文传输，存在安全风险的问题。

HTTPS 则解决 HTTP 不安全的缺陷，在 TCP 和 HTTP 网络层之间加入了 SSL/TLS 安全协议，使得报文能够加密传输。

HTTP 连接建⽴相对简单，TCP 三次握⼿之后便可进行 HTTP 的报⽂传输。

而HTTPS 在 TCP 三次握⼿之后，还需进⾏ SSL/TLS 的握⼿过程，才可进⼊加密报⽂传输。

两者的默认端⼝不⼀样，HTTP 默认端⼝号是 80，HTTPS 默认端⼝号是 443。

**机制**

HTTPS 协议需要向 CA（证书权威机构）申请数字证书，来保证服务器的⾝份是可信的。

https 通过混合加密（对称⾮对称）保证信息的机密性、摘要算法保证数据的完整性，将服 务 器的公钥放⼊数字证书中，防⽌冒充

#### https、 rsa 和aes加密介绍，前端加密和https加密区别，前端加密具体实现思路与场景

**HTTPS、RSA和AES加密介绍**

**HTTPS**

HTTPS（超文本传输安全协议）是HTTP协议的安全版本，它在传输数据时使用SSL（安全套接层）或TLS（传输层安全）协议进行加密。HTTPS的主要目的是在互联网上提供安全可靠的通信方式，保护数据隐私和安全性，防止敏感信息被窃取、篡改和劫持。

**RSA**

RSA是一种非对称加密算法，它使用一对密钥（公钥和私钥）来进行加密和解密。公钥用于加密数据，而私钥用于解密。RSA算法的安全性基于大数分解的困难性，广泛用于数据传输的安全领域，如数字签名和密钥交换。

**AES**

AES（高级加密标准）是一种对称加密算法，它使用相同的密钥进行加密和解密。AES算法在速度和安全性方面表现优异，被广泛应用于各种加密场景，如存储设备加密、通信加密等。

**前端加密与HTTPS加密的区别**

**前端加密**

前端加密通常指的是在客户端（用户浏览器）进行的加密操作。它的主要目的是保护用户输入的数据，如表单数据、密码等，在传输到服务器之前进行加密，以防止数据在传输过程中被截获。

**HTTPS加密**

HTTPS加密则是在传输层进行的，它保护了整个客户端和服务器之间的通信过程。HTTPS不仅加密了数据本身，还提供了数据完整性和身份验证，确保数据在传输过程中不被篡改，并且通信双方的身份是可验证的。

**前端加密的具体实现思路与场景**

**实现思路**

1. **使用JavaScript库**：可以使用现有的JavaScript加密库，如CryptoJS，来实现前端加密。
2. **加密敏感数据**：在用户输入敏感信息（如密码）时，使用JavaScript进行加密，然后再发送到服务器。
3. **安全存储密钥**：确保加密密钥在前端的安全存储，避免泄露。

**应用场景**

1. **表单数据加密**：在用户提交表单时，对敏感信息进行加密，保护用户隐私。
2. **客户端存储加密**：在本地存储（如localStorage）中加密存储数据，防止数据被未授权访问。
3. **保护API通信**：在调用API时，对请求数据进行加密，增加数据安全性。

综上所述，前端加密和HTTPS加密各有其应用场景和优势，通常建议结合使用，以提供更全面的数据保护措施。

#### TSL加密过程

完整加密流程(以TLS 1.3为例，简化版)

##### 步骤1:握手协商(安全交换密钥)

* 客户端发起请求:客户端向服务器发送“客户端问候”，包含支持的TLS版本、加密套件(如ECC+AES)、随机数A
* 服务器身份验证:服务器回复“服务器问候”，确认TLS版本和加密套件，发送随机数B，并附上数字证书(由CA机构签发，含服务器公钥
* 客户端验证证书:客户端校验证书合法性(是否过期、是否被CA信任)，通过后从证书中提取服务器公钥
* 生成会话密钥:客户端用服务器公钥加密“预主密钥”(结合随机数A/B生成)发送给服务器;服务器用自己的私钥解密，得到预主密钥
* “预主密钥+随机数A+随机数B”，通过相同算法生成对称会话密钥(后续数据加密用)

##### 步骤2:会话加密(传输实际数据)

* 握手完成后，客户端和服务器用协商好的对称密钥(如AES)加密所有通信数据，同时用哈希算法(如SHA-256)保证数据完整性(防止被篡改)
* 通信结束后，会话密钥会被销毁，下次通信需重新握手生成新密钥

#### 死锁

死锁是并发编程中一组进程或线程因争夺系统资源而相互等待，导致所有进程都无法继续执行的状态，理解死锁并掌握其应对方法，对开发稳定可靠的程序很重要。

| **必要条件** | **含义** |
| :--- | :--- |
| **互斥条件** | 资源同时只能被一个进程或线程使用，其他请求者只能等待 |
| **请求与保持条件** | 进程在等待新资源的同时，不会释放已持有的资源 |
| **不可剥夺条件** | 进程已获得的资源，在使用完之前不能被强制剥夺，只能由该进程主动释放 |
| **循环等待条件** | 多个进程或线程之间形成了一种头尾相接的循环等待资源的关系 |

##### 解决死锁的方法

解决死锁主要有以下四种策略，下表概括了它们的特点和适用场景，你可以根据实际需要选择。

| **策略** | **核心思想** | **适用场景** | **优点** | **缺点** |
| :--- | :--- | :--- | :--- | :--- |
| **预防死锁** | 破坏死锁四个必要条件中的至少一个 | 高可靠性要求的系统（如银行、航空控制系统） | 从根源上避免死锁发生，较为彻底  | 可能限制系统并发性和资源利用率，影响性能 |
| **避免死锁** | 在资源分配前动态检查，确保系统始终处于安全状态  | 资源有限的系统（如操作系统、数据库管理系统）  | 比预防死锁更灵活，资源利用率较高  | 实现复杂，需要预先知道进程的最大资源需求，有计算开 |
| **检测与解除** | 允许死锁发生，但会定期检测，并在发现后采取措施解除  | 死锁发生频率较低，或采用其他策略成本过高的系统  | 无需限制进程执行和资源申请，资源利用率高  | 检测和恢复机制的实现开销较大，进程终止或回滚可能带来代价  |
| **鸵鸟策略** | 忽略死锁问题，假装它不会发生或不重要 | 死锁发生频率极低，或处理死锁的成本远超其影响的系统（如一些桌面应用） | 实现简单，没有额外性能开销 | 死锁一旦发生，可能导致程序长时间无响应或卡死，需用户手动干预 |

##### 预防死锁

通过**破坏死锁的四个必要条件之一**来防止死锁发生

***破坏互斥条件**：让资源能够同时被多个进程访问。但这通常难以实现，因为很多资源（如打印机、变量锁）本质就是互斥的
***破坏请求与保持条件**：要求进程在开始执行前**一次性申请其所需的所有资源**。如果无法全部满足，则需等待。这可能导致资源浪费和饥饿
***破坏不可剥夺条件**：允许系统在特定条件下**强制剥夺**进程已获得的资源。实现复杂，可能造成前功尽弃
***破坏循环等待条件**：采用**资源有序分配法**（或称为“锁排序”）。为所有资源类型规定一个全局的申请顺序，所有进程都按此顺序申请资源，这是**最实用**的方法之一

##### 避免死锁

在资源分配过程中**动态地检查**此次分配是否会导致系统进入不安全状态，从而避免死锁

**银行家算法（Banker's Algorithm）**：一个经典的死锁避免算法

其核心思想是：在分配资源前，系统会**模拟**此次分配，并检查模拟后系统是否仍处于**安全状态**（即存在一个序列能使所有进程顺利完成）。只有在安全状态下才会实际分配资源**69**。

##### 检测与解除

如果系统允许死锁发生，就需要**定期检测**死锁，并在检测到后**解除**它

**死锁检测**：通常通过构建**资源分配图**并检查图中是否存在**环路**来实现

**死锁解除**：一旦检测到死锁，可采取以下措施：

**终止进程**：强制终止一个或多个死锁进程以打破循环

**资源剥夺**：强制剥夺一个或多个死锁进程占用的部分资源，再将这些资源分配给其他进程。被剥夺资源的进程可能需要回滚到之前某个安全状态

#### dns解析[baidu.com](https://baidu.com)和[google.com](https://google.com)的不同之处

DNS解析过程对于不同的域名基本上是相似的，但是解析[baidu.com](https://baidu.com)和[google.com](https://google.com)这样的不同域名可能会有以下不同之处：

1. **DNS服务器响应**：
   * [baidu.com](https://baidu.com)的DNS解析可能会被导向百度自己的DNS服务器或者与其合作的服务商。
   * [google.com](https://google.com)的DNS解析可能会被导向谷歌的全球DNS服务器，如Google Public DNS。
2. **地理位置和CDN**：
   * 由于百度主要服务于中国市场，解析[baidu.com](https://baidu.com)时，DNS服务器可能会根据用户的地理位置返回最近的服务器地址，这些服务器可能主要位于中国境内。
   * 谷歌的服务是全球性的，因此解析[google.com](https://google.com)时，可能会返回全球范围内的多个数据中心地址，这些数据中心分布在不同国家和地区。
3. **DNS污染和劫持**：
   * 在某些国家和地区，由于网络审查的原因，[google.com](https://google.com)的DNS解析可能会受到污染或劫持，导致用户无法访问或被重定向到其他网站。
   * [baidu.com](https://baidu.com)作为国内的搜索引擎，通常不会受到类似的DNS污染或劫持。
4. **解析速度和稳定性**：
   * [baidu.com](https://baidu.com)的DNS解析可能会更快更稳定，因为它主要针对中国用户进行了优化。
   * [google.com](https://google.com)的DNS解析速度和稳定性可能会因地理位置和网络环境的不同而有所差异。
5. **IPv6支持**：
   * [google.com](https://google.com)通常会提供IPv6地址作为解析结果之一，因为谷歌大力推动IPv6的普及。
   * [baidu.com](https://baidu.com)也可能提供IPv6支持，但普及程度可能不如谷歌。
6. **安全性和隐私**：
   * 谷歌的DNS服务器（如8.8.8.8和8.8.4.4）注重用户隐私保护，不会记录用户的IP地址。
   * 百度的DNS服务器可能会有不同的隐私政策，用户应当查看相关的服务条款。
7. **DNSSEC**：
   * [google.com](https://google.com)支持DNSSEC（DNS安全扩展），这是一种增加DNS查询安全性的技术。
   * [baidu.com](https://baidu.com)可能也支持DNSSEC，但具体实施情况可能会有所不同。

#### 了解哪些网络通讯协议（选背）

网络通讯协议是计算机网络中的规则集合，它们定义了数据交换的格式、消息传递的规则、连接的建立和终止过程等。以下是一些常见的网络通讯协议：

1. **HTTP (Hypertext Transfer Protocol)**: 互联网上应用最广泛的协议之一，用于在Web服务器和客户端之间传输超文本数据。
2. **HTTPS (HTTP Secure)**: HTTP的安全版本，通过SSL/TLS加密，提高了数据传输的安全性。
3. **TCP (Transmission Control Protocol)**: 提供面向连接的、可靠的数据传输服务。
4. **UDP (User Datagram Protocol)**: 提供无连接的数据传输服务，不保证数据的可靠传输。
5. **IP (Internet Protocol)**: 负责将数据包传送给目标计算机，是互联网的基础协议。
6. **ICMP (Internet Control Message Protocol)**: 用于发送控制消息，比如ping命令就是使用ICMP协议。
7. **SMTP (Simple Mail Transfer Protocol)**: 用于电子邮件的发送。
8. **IMAP (Internet Message Access Protocol)**/ **POP3 (Post Office Protocol version 3)**: 用于电子邮件的接收。
9. **FTP (File Transfer Protocol)**: 用于文件的传输。
10. **SFTP (Secure File Transfer Protocol)**/ **FTPS (FTP Secure)**: 安全的文件传输协议，提供了加密的文件传输。
11. **SSH (Secure Shell)**: 用于安全地访问远程计算机。
12. **DNS (Domain Name System)**: 将域名解析为IP地址。
13. **DHCP (Dynamic Host Configuration Protocol)**: 自动为网络上的设备分配IP地址。
14. **NTP (Network Time Protocol)**: 用于网络时间同步。
15. **SSL/TLS (Secure Sockets Layer / Transport Layer Security)**: 为网络通信提供安全及数据完整性的一种安全协议。
16. **WebSocket**: 提供全双工通信渠道，允许服务器和客户端之间实时通信。
17. **MQTT (Message Queuing Telemetry Transport)**: 用于物联网设备间的轻量级通信协议。
18. **AMQP (Advanced Message Queuing Protocol)**: 用于消息中间件的通信协议。
19. **RPC (Remote Procedure Call)**: 允许程序像调用本地程序一样调用远程程序。
20. **SOAP (Simple Object Access Protocol)**: 基于XML的协议，用于Web服务之间的通信。

这些协议各有用途，它们共同构成了互联网和各种网络应用的基础。随着技术的发展，新的协议也在不断出现，以适应不同的网络需求。

#### 协商缓存有无过期时间、有哪些标记位

**一、协商缓存的定义与作用**

协商缓存（也称为条件请求缓存）允许客户端在缓存过期后，向服务器发送带有条件请求头的验证请求，服务器根据资源是否更新决定是否返回完整资源或304状态码**27**。这种方式既减少了重复传输，又能确保用户获取最新数据。

**二、协商缓存的工作流程**

1. **触发条件**：当强缓存（如`Cache-Control`）失效或不存在时，浏览器会触发协商缓存**24**。
2. **携带验证头**：客户端在请求中附加`If-None-Match`或`If-Modified-Since`等字段，分别对应服务器响应中的`ETag`或`Last-Modified`**7**。
3. **服务器验证**：服务器比较请求头与资源的当前状态：
   * 若资源未修改，返回`304 Not Modified`，客户端继续使用缓存**2**。
   * 若资源已修改，返回完整资源及新的验证头**7**。

**三、协商缓存与强缓存的区别**

***强缓存**：直接使用本地缓存，不请求服务器，通过`Cache-Control`或`Expires`控制**46**。
***协商缓存**：需与服务器交互，但仅传输少量验证头，适合动态内容或需实时性的场景**2**。

**四、实际应用场景**

***静态资源**：通常结合强缓存与协商缓存，先通过强缓存快速加载，过期后通过协商缓存验证更新**48**。
***动态内容**：如API数据，使用协商缓存确保数据实时性，同时减少带宽消耗。

**一、协商缓存的标记位**

1. **Last-Modified 和 If-Modified-Since**
   * **Last-Modified**：服务器在响应头中返回资源的最后修改时间**3**。
   * **If-Modified-Since**：客户端在请求头中携带此字段，值为之前缓存的 Last-Modified 时间。服务器比较该时间与资源的当前修改时间，若未修改则返回 304 状态码**3**。
2. **ETag 和 If-None-Match**
   * **ETag**：服务器为资源生成的唯一标识符（如哈希值），用于更精确地判断资源是否变更**3**。
   * **If-None-Match**：客户端携带缓存的 ETag 发送请求，服务器对比后决定是否返回新资源或 304 状态码**3**。

**二、协商缓存与强缓存的关系**

协商缓存仅在强缓存失效时触发，例如当 `Cache-Control` 的 `max-age` 过期或 `Expires` 超时后，浏览器才会携带上述标记位向服务器发起验证请求**&#x33;5&#x36;**。

**三、实际应用中的注意事项**

***优先级**：ETag 比 Last-Modified 更精确，但 Last-Modified 兼容性更好。
***性能影响**：协商缓存需额外请求头，但可减少带宽消耗，适合动态内容或需实时更新的场景**5**。

通过上述标记位，协商缓存实现了资源更新状态的高效验证，平衡了性能与数据一致性。更多细节可参考相关技术文档**3**。

#### lastmodified未变化请求一次还是两次

***初始请求**：浏览器首次请求资源时，服务器返回状态码 200，并在响应头中包含 `Last-Modified`，标记资源的最后修改时间**13**。
***后续请求**：浏览器再次请求同一资源时，会在请求头中携带 `If-Modified-Since`，值为之前缓存的 `Last-Modified` 时间**27**。
***服务器验证**：服务器比较 `If-Modified-Since` 与资源的当前修改时间。若未变化，返回状态码 304（Not Modified），不返回资源内容；若变化，则返回完整资源（状态码 200）

#### 协商缓存返回的状态码，lastmodified和etag的区别

**1. Last-Modified 与 ETag 的作用**

***Last-Modified**：标记资源的最后修改时间。浏览器首次请求资源时，服务器在响应头中返回 `Last-Modified`；再次请求时，浏览器通过 `If-Modified-Since` 头携带该时间，服务器比较后决定是否返回完整资源或 `304 Not Modified`**67**。
***ETag**：资源的唯一标识符（如哈希值）。服务器首次请求时返回 `ETag`，后续请求通过 `If-None-Match` 头携带该值，服务器验证后返回相应结果**67**。

**2. 核心区别**

***精度**：
  * Last-Modified 仅精确到秒级，若资源在1秒内多次修改，可能无法准确识别变化**67**。
  * ETag 是唯一标识符，能更精确地判断资源是否变化，适用于高频修改的场景**67**。
***性能**：
  * Last-Modified 仅需比较时间戳，计算开销小**6**。
  * ETag 需生成哈希值，服务器负担较重，但安全性更高**67**。
***兼容性**：
  * Last-Modified 是 HTTP 标准字段，兼容性更好**6**。
  * ETag 可能因服务器集群或负载均衡导致不一致问题（如不同服务器生成不同 ETag）**7**。

**3. 协商缓存流程**

***Last-Modified 流程**：
  1. 首次请求：服务器返回 `Last-Modified` 头**67**。
  2. 再次请求：浏览器携带 `If-Modified-Since`，服务器比较后返回 `200`（更新）或 `304`（未修改）**67**。
***ETag 流程**：
  1. 首次请求：服务器返回 `ETag` 头**67**。
  2. 再次请求：浏览器携带 `If-None-Match`，服务器比较后返回相应状态码**67**。

**4. 使用场景建议**

* 优先使用 **ETag**：适用于需要高精度判断资源变化的场景（如动态内容）**67**。
* 结合 **Last-Modified**：适用于静态资源或对性能要求较高的场景

#### Etag底层怎么判断两个文件是否相同

**Etag的生成原理**

***静态文件**：在如Nginx或Apache等服务器中，Etag通常由响应头的`Last-Modified`和`Content-Length`组合而成，并将其转换为十六进制格式。例如，Nginx会将这两个字段合并后生成Etag值
***动态内容**：对于动态生成的文件（如Koa框架中的静态文件），Etag可能基于文件内容的哈希值或其他唯一标识符生成

**判断文件是否相同的流程**

***首次请求**：服务器生成Etag并返回给客户端，通常作为响应头的一部分
***再次请求**：客户端在后续请求中通过`If-None-Match`头携带该Etag值，服务器收到后与当前资源的Etag进行比较：
  * 如果相同，返回`304 Not Modified`，表示资源未变化。
  * 如果不同，返回完整资源和新的Etag，表示资源已更新

**实际应用场景**

***静态资源**：如图片、CSS等，通常通过`Last-Modified`和`Content-Length`生成Etag，平衡性能与准确性
***动态内容**：如JSON或XML响应，可能基于内容哈希生成Etag，确保唯一性

#### 强缓存的头信息介绍

**1. Cache-Control**

***定义**：`Cache-Control` 是HTTP/1.1引入的强缓存头字段，用于指定缓存指令，如最大存活时间（`max-age`）、是否允许缓存（`public`/`private`）等**78**。
***常见指令**：
  * `max-age=秒数`：表示资源在缓存中有效的最长时间，例如 `max-age=3600` 表示缓存1小时后失效**79**。
  * `public`：允许所有缓存（如CDN、浏览器）存储资源。
  * `private`：仅允许浏览器端缓存，中间代理（如CDN）不可缓存**7**。
***优先级**：高于`Expires`，是现代HTTP缓存的推荐字段**9**。

**2. Expires**

***定义**：`Expires` 是HTTP/1.0的字段，指定资源过期的绝对时间（如 `Expires: Wed, 21 Oct 2025 07:28:00 GMT`）**34**。
***局限性**：依赖客户端时间，若本地时间与服务器不同步可能导致缓存失效**9**。

**3. 强缓存的工作流程**

* 首次请求资源时，服务器返回`Cache-Control`或`Expires`头，浏览器将资源存入缓存**9**。
* 再次请求时，浏览器检查缓存是否过期：
  * 若未过期，直接使用缓存（状态码200 from cache）**911**。
  * 若过期，则进入协商缓存流程（如对比`ETag`或`Last-Modified`）**911**。

**4. 与协商缓存的区别**

* 强缓存完全依赖本地判断，不与服务器交互；协商缓存需通过`ETag`或`If-Modified-Since`等头与服务器验证资源是否更新

#### 粘包

##### 什么是粘包？

粘包指的是在数据传输过程中，发送方发送的多个数据包在接收方被合并成一个数据包接收，或者一个数据包被拆分成多个部分接收，导致接收方无法正确区分每个独立的消息边界。

##### 产生的原因

1. **TCP 是面向字节流的协议**

TCP 不保留消息之间的边界，它只保证数据按顺序、可靠地传输。因此，多个 send 操作可能被合并成一个 TCP 段发送，而一个 send 也可能被拆分成多个 TCP 段。

2. **Nagle 算法的影响**

TCP 默认启用 Nagle 算法，它会将多个小的数据包合并成一个大的数据包发送，以提高网络效率。这也会导致粘包现象

3. **接收方缓冲区机制**

接收方从缓冲区读取数据时，可能一次性读取了多个发送包的数据，也可能只读取了一个包的一部分，造成消息边界混乱

##### 粘包的典型场景

* 短时间内频繁发送小包（如聊天消息、心跳包）
* 高并发下，多个客户端同时发送数据，服务器接收时发生粘包
* 使用非阻塞 I/O 或异步 I/O 时，读取不完整

##### 如何解决粘包问题？

* 固定长度消息*

每条消息使用固定长度，接收方按固定长度读取。例如每条消息 100 字节，不足则补空格或 0

**缺点**：浪费带宽，不够灵活。

**

* 特殊分隔符*

在每条消息末尾添加特殊分隔符（如 `\n` 或 `\0`），接收方根据分隔符拆分消息。

**缺点**：如果消息体中包含分隔符，则需转义处理。

**

* 消息头 + 消息体（Length-Prefixed）*

每条消息前附加一个固定长度的头部，表示消息体的长度。接收方先读取头部，再根据长度读取消息体。

**优点**：高效、可靠，是业界常用方案。

* 应用层协议设计*

使用成熟的应用层协议（如 HTTP、WebSocket、Protobuf、gRPC 等），它们自带消息边界机制，避免粘包问题

#### keep-alive的长连接机制、keepalive复用tcp链接最多支持多少个协议

1. Keep-Alive 的工作原理

* 协商机制：客户端通过请求头 `Connection: keep-alive` 声明支持持久连接。服务器通过响应头 `Connection: keep-alive` 表示同意复用连接，并通过 `Keep-Alive: timeout=5, max=100` 指定连接的超时时间和最大请求数。

2. 连接生命周期：客户端和服务器保持连接打开，直到满足以下条件之一：

* 达到 `max` 指定的最大请求数。
* 超过 `timeout` 指定的空闲时间（秒）。
* 任意一方主动关闭连接（如发送 `Connection: close`）。

3. Keep-Alive 的优势

* 减少连接开销：避免频繁建立和关闭 TCP 连接的开销（三次握手和四次挥手），从而降低延迟和资源消耗。
* 降低延迟：复用连接避免了重复建立连接的时间开销，尤其在高延迟网络中，能够显著加快页面加载速度。
* 提升吞吐量：在同一个连接上，客户端可以按顺序或并行发送多个请求。HTTP/1.1 支持管道化，但存在队头阻塞（Head-of-Line Blocking）问题；而 HTTP/2 进一步优化为多路复用，彻底解决了这一问题。

4. Keep-Alive 的底层原理

* TCP 层的 Keep-Alive 机制：TCP 协议本身支持 Keep-Alive 机制，用于检测连接状态。当 TCP 连接空闲一段时间后，会发送一个 Keep-Alive 探测包（无意义的字节），以确认对方是否仍然处于连接状态。如果对方回复了确认包，就表明连接仍然有效；否则，连接会被关闭。
* 配置参数：

`tcp_keepalive_time`：连接空闲时间（默认 7200 秒，即 2 小时）。

`tcp_keepalive_intvl`：探测包的间隔时间（默认 75 秒）。

`tcp_keepalive_probes`：最大探测次数（默认 9 次）。

5. Keep-Alive 的实际应用

* Web 浏览器：加载网页时会并发请求多个资源（如图片、CSS、JS），长连接可极大提升加载速度。
* API 接口：高频调用时减少连接建立/关闭的开销。
* 反向代理/负载均衡：通常会配置长连接以提升性能。
